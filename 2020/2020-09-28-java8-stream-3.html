深入理解 Java 8 的 Stream 流式编程（三）

kun 2020-09-28 21:30:00 StreamJava 8

关于 Java 8 的 Stream 流式编程内容较多，所以日志分为三部分：

  1. 常用的 Stream API：对于新东西我一般先学会用，然后再慢慢的深入了解。
  2. 关于 Straem 的一些总结：熟练使用之后就该总结一下了。
  3. Stream 的原理：知道工具的原理才能更好的使用工具。

这篇是该日志的第三部分，也就是一些关于 Stream 背后原理的总结。

# 流是怎么执行的

首先我们先思考一个问题，对一个流进行多步操作时，每一步是怎么执行的？每一步都是把流中的元素都循环一遍吗？

我们先看一段示例代码：

    String[] strs = new String[]{"AAA","bbbb","ccc","DDddD"};
    OptionalInt max = Arrays.stream(strs)
            .peek(System.out::println)
            .filter(s -> s.length() > 3)
            .peek(System.out::println)
            .mapToInt(String::length)
            .peek(System.out::println)
            .max();
    System.out.printf("字符串最大长度是：%d%n", max.getAsInt());

这段代码的逻辑是：

  1. 对一个字符串数组进行遍历打印，
  2. 过滤出长度大于 3 的字符串
  3. 再打印一遍过滤后的数组
  4. 将过滤后的数组转换成长度组成的int数组
  5. 提取出这个 int 数组的对大值

执行结果如下：

AAA
bbbb
bbbb
4
ccc
DDddD
DDddD
5
字符串最大长度是：5

从执行结果来看，很明显中单步操作并不是把整个数组遍历一遍然后再执行下一步操作，而是数组中的每个元素依次通过每一步操作。 这也印证了前一篇中总结的流的特点：只遍历一遍。 想想也是啊，如果每一步操作都要遍历整个数组，那一系列操作下来要遍历 n 遍数组，性能肯定无法接受。

总结一下就是，对流的操作像是一个通道，每一步操作都是这个通道上的一节管道（Pipeline），流中的每一个元素依次通过管道，最后出来的就是最终的结果。 每节管道就叫 Stream Pipelines。

# Stream Pipelines

把对流的操作抽象成 Pipeline 就会有一下几个问题：

# 每一步操作是怎么封装成一节管道（Pipeline）

下面这张图是跟 Pipeline 相关的接口和类的继承关系图，还有 IntPipeline, LongPipeline, DoublePipeline 没在图中画出， 这三个类专门为三种基本类型（不是包装类型）而定制的，跟 ReferencePipeline 是并列关系。 Head用于表示第一个 Pipeline，即调用调用诸如Collection.stream()方法产生的 Pipeline， 很显然这个 Pipeline 里不包含任何操作；StatelessOp 和 StatefulOp 分别表示无状态和有状态的 Pipeline， 对应于无状态和有状态的中间操作。

# 管道的操作怎么串联起来

每一个 Pipeline 的关联方式类似一个双向连表，由于每个 Pipeline 都记录了前一个 Pipeline 和本次的操作以及回调函数， 依靠这种结构就能建立起对数据源的所有操作。而把这些 Pipeline 关联起来的一个关键接口就是 java.util.stream.Sink 接口。

Sink 接口有这样四个方法：

begin(long size)：开始遍历元素之前调用该方法，通知 Sink 做好准备。

end()：所有元素遍历完成之后调用，通知 Sink 没有更多的元素了。

cancellationRequested()：是否可以结束操作，可以让短路操作尽早结束。

accept(T t)：遍历元素时调用，接受一个待处理元素，并对元素进行处理。 Pipeline 把自己包含的操作和回调方法封装到该方法里，前一个 Pipeline 只需要调用当前 Pipeline 的 accept(T t)方法就行了。

有了 Sink 接口，Pipeline 之间的串联就很简单了，前一个 Pipeline 再执行完自己的 accept 方法后执行后一个 Pipeline 的 accept 方法就行了， 不用去关心后一个 Pipeline 是怎么实现的。

# 总结

Stream API 架构是想当复杂的，理解它的基本原理有利于我们写出更高效的代码。

  • 每一步操作是怎么封装成一节管道（Pipeline）
  • 管道的操作怎么串联起来
